Executive summary

The catalog tells one dominant story: Indiana University is teaching its institutional data to answer questions in plain language, and the hard part every team is really solving is trust — making sure a plain-language answer is correct and never leaks protected data. Three teams arrived at strikingly similar architectures independently, and a shared on-prem platform, REALLMS, is emerging as the connective tissue between them.

  • ▪  One problem, three teams, converging answers. , and (all IA-RA), (EDED), and (UITS-SSA) are all variations on “ask a question, get a governed answer.”
  • ▪  REALLMS is the de facto standard. Seven of nine solutions use or plan to use it, spanning all three teams. It is the privacy anchor that keeps institutional data on IU infrastructure.
  • ▪  The shared trust strategy is metadata-grounding, not data-exposure. , , and all feed the model schema, definitions and examples while keeping actual records out of the model.
  • ▪  EDED has built the most mature trust stack; IA-RA has the most use cases. That asymmetry is the single clearest reuse opportunity in the catalog.
  • ▪  Governance is designed in, not bolted on — the Data Cookbook / DM-01 classification (, ) and Denodo row/column security ().
  • ▪  A second, smaller theme is LLM thematic coding of qualitative text — and — today an IA-RA specialty.
  • ▪  One deliberate outlier. is the only solution with no LLM and no REALLMS — a rapid Python/Jupyter build. A useful reminder that not every data question needs a language model.

Deep dive — translating plain language into trustworthy data results

This is the through-line of the catalog. The temptation with a large language model is to let it answer questions about data directly; the danger is that it will confidently return a wrong number, and in institutional reporting a wrong number presented as fact is worse than no answer at all. What is interesting is that, without a shared playbook, the teams converged on the same defenses. Read together, their projects form a “trust ladder” — each rung constrains the model further so the final answer can be relied upon.

  1. Ground the model in metadata, never the data. Feed it schema, table relationships, Data Cookbook definitions and real column values — but not records. does this with a RAG pipeline; grounds on the full 152-column schema, Data Cookbook and business rules and states plainly that it “never sees actual records.”
  2. Retrieve verified exemplars before generating. checks for an exact or near match against 120+ curated question→SQL pairs first, so common questions never depend on the model improvising.
  3. Validate and auto-repair the output deterministically. validates generated Oracle SQL against the rules and auto-repairs (up to twice) before the query ever reaches a human — a non-LLM safety net around an LLM.
  4. Keep a human checkpoint at the trust boundary. is deliberately not wired to live data yet: it hands back ready-to-run SQL for a person to execute, rather than fabricating a result set. That design choice is a trust statement.
  5. Close the loop so accuracy compounds. A “save as training pair” feedback loop () turns every expert correction into a new verified example, so the system gets more reliable the more it is used.
  6. Enforce governance so protected rows can never leak. ’s fail-closed gate classifies every referenced column and routes sensitive data to an on-prem reader; the pushes row/column security, masking and audit into the data layer itself.
  7. Prefer grounded, explainable answers over fluent ones. and both frame their goal as grounded summaries and comparisons rather than free-form generation.

The read: the pilot question posed by — “can an LLM reliably turn plain language into correct SQL?” — has effectively been answered by : yes, if you wrap the model in grounding, verified examples, deterministic validation and a human checkpoint. The recommendation is to treat NaLQi’s full stack as IU’s reference pattern for trustworthy NL→data, paired with Crimson Curator’s gate and the Denodo rooms for governance.

Deep dive — the role of REALLMS

REALLMS is the platform that shows up in seven of nine solutions and across all three teams — the closest thing the catalog has to a standard. From the project data, it is IU’s on-prem LLM platform: it runs inside the IU data center ( names Qwen3-Coder-Next for generation and Qwen3-Embedding-8B for retrieval) and exposes APIs teams build against. Its strategic value is simple: it keeps institutional data on IU infrastructure, which is what makes FERPA- and HIPAA-aligned AI use possible.

What is striking is how many different jobs it does:

  • ▪  Generation engine for natural-language-to-SQL — and .
  • ▪  Embeddings / retrieval for semantic example matching — .
  • ▪  Coding API for qualitative thematic analysis — .
  • ▪  On-prem “safe reader” of sensitive rows that returns only a scrubbed answer, so frontier models never see protected data — .
  • ▪  Metadata curator that decides which columns to keep, drop or alias and writes self-describing view tags — .
  • ▪  Target migration platform for solutions prototyped elsewhere — (from Microsoft Copilot) and (from NotebookLM).

Adoption spans the maturity curve — already in use (), demo-ready (, ), in development (), and planned or exploratory (, ).

The read: REALLMS is already the shared backbone, but each team integrates with it independently. The most valuable architectural move visible in the data is ’s hybrid split — frontier models for public data, REALLMS for anything sensitive — which lets IU use the best available models without moving protected data off-premises. Pairing a common REALLMS integration layer with the Denodo curated rooms and a shared governance gate would turn seven independent integrations into one reusable platform.

NaLQi — Natural Language Query Intelligence

Team: UITS Enterprise Data Services (EDED)  ·  Status: In development · demo-ready (not wired to live data yet, by design)

A standalone, privacy-first application that translates plain-English questions into correct, ready-to-run Oracle SQL for IU’s Census enrollment data (152 columns across demographics, academic programs, campuses, and terms). The model is grounded in the real schema, Data Cookbook definitions, business rules, and a curated library of 120+ verified question→SQL examples — it never sees actual records. Runs entirely on IU infrastructure using REALLMS (Qwen3-Coder-Next for generation, Qwen3-Embedding-8B for retrieval); a governed data-access layer (Denodo / Oracle) is the planned path to live results.

Denodo AI Chat Room — LLM-Ready Governed Views

Team: UITS Enterprise Data Services (EDED)  ·  Status: In development

An infrastructure-level approach that creates curated, LLM-facing views inside IU’s governed Denodo platform (iu_ia_dev). Instead of training an external app on schema knowledge, it embeds that knowledge into the data layer itself: a Python curation pipeline reads source views and column metadata, sends them to REALLMS for LLM-driven curation (deciding which columns to keep, drop, or alias for clarity), resolves keys and associations, and emits VQL to create a self-describing view such as enrollment_room — complete with a structured DOMAIN tag describing the view’s purpose, persona, and how to query it. Because the views live in Denodo, they inherit row- and column-level security, masking, and centralized audit logging (HIPAA-aligned), so any AI agent — including NaLQi — can consume them safely. The model works only on metadata; no data records are ever sent to the LLM.
